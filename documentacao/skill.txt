Orientar um agente de programação na implementação completa do MVP 1 do PueriCare, um protótipo clínico para acompanhamento de crianças do nascimento até 5 anos e 11 meses.
A skill deve garantir:
•	escopo controlado;
•	arquitetura simples;

•	validação por testes;

2. Contexto do produto
O PueriCare é um sistema de apoio ao registro de consultas de puericultura.
A faixa etária inicial é:
nascimento até 5 anos e 11 meses
O produto deve permitir que um profissional:
•	cadastre crianças;
•	registre consultas;
•	acompanhe crescimento;
•	registre marcos do desenvolvimento;
•	consulte uma base científica por meio de IA;
•	gere relatórios editáveis;
•	exporte relatórios em PDF.
O MVP é um protótipo funcional. Não deve tentar resolver toda a complexidade clínica da puericultura.
________________________________________
3. Princípio central
O agente deve sempre seguir esta regra:
Implementar apenas o necessário para validar o uso clínico do protótipo.
Não adicionar módulos, integrações ou automações fora do escopo sem autorização explícita.

10. Regras de segurança
Todas as coleções clínicas devem conter ownerId.
O usuário autenticado só poderá:
•	criar documentos com seu próprio ownerId;
•	ler documentos cujo ownerId seja igual ao seu UID;
•	editar apenas seus próprios documentos;
•	excluir logicamente seus próprios documentos.
Exemplo conceitual:
request.auth != null
AND resource.data.ownerId == request.auth.uid
Para criação:
request.resource.data.ownerId == request.auth.uid
O agente deve criar e testar as regras do Firestore antes de considerar o módulo concluído.

11. Regras funcionais por módulo
11.1 Autenticação
Implementar:
•	login com e-mail e senha;
•	logout;
•	recuperação de senha;
•	proteção de rotas;
•	redirecionamento para login quando não autenticado;
•	persistência da sessão.
Critérios:
•	usuário não autenticado não acessa páginas internas;
•	mensagens de erro são compreensíveis;
•	senha nunca é armazenada manualmente.
________________________________________
11.2 Dashboard
Exibir:
•	total de crianças ativas;
•	total de consultas;
•	consultas mais recentes;
•	botão para cadastrar criança;
•	botão para iniciar consulta.
O dashboard não precisa ter gráficos avançados.
________________________________________
11.3 Cadastro da criança
Campos obrigatórios:
•	nome;
•	data de nascimento;
•	sexo;
•	nome do responsável.
Campos opcionais:
•	telefone;
•	idade gestacional;
•	tipo de parto;
•	peso ao nascer;
•	comprimento ao nascer;
•	perímetro cefálico ao nascer.
Validações mínimas:
•	data de nascimento não pode ser futura;
•	criança deve ter até 5 anos e 11 meses no momento do cadastro;
•	peso e medidas devem aceitar apenas valores plausíveis;
•	nome deve ser normalizado antes de salvar.
Excluir deve significar exclusão lógica por deletedAt.
________________________________________
11.4 Lista de pacientes
Implementar:
•	listagem;
•	busca por nome;
•	ordenação alfabética;
•	acesso ao perfil;
•	edição;
•	exclusão lógica;
•	ação para iniciar consulta.
Não implementar paginação complexa no MVP, salvo necessidade técnica.
________________________________________
11.5 Consulta
A consulta deve ser realizada em uma única tela ou formulário simples.
Campos:
•	data da consulta;
•	idade calculada automaticamente;
•	alimentação;
•	sono;
•	situação vacinal;
•	observações clínicas;
•	conduta;
•	medidas de crescimento;
•	checklist de desenvolvimento.
A consulta deve permitir:
•	salvar rascunho;
•	editar;
•	concluir;
•	gerar relatório.
Não implementar protocolo clínico complexo.
________________________________________
11.6 Crescimento
Registrar:
•	peso;
•	altura ou comprimento;
•	perímetro cefálico;
•	data da medição.
Mostrar:
•	histórico em tabela;
•	gráfico temporal simples;
•	unidades visíveis.
Regras:
•	peso em quilogramas;
•	altura e perímetro cefálico em centímetros;
•	não permitir valores negativos;
•	não interpretar automaticamente curvas ou diagnósticos.
O gráfico deve mostrar apenas evolução temporal neste MVP.
________________________________________
11.7 Desenvolvimento
Os marcos devem ser selecionados conforme a idade da criança.
Faixas sugeridas:
•	0–2 meses;
•	3–5 meses;
•	6–8 meses;
•	9–11 meses;
•	12–17 meses;
•	18–23 meses;
•	24–35 meses;
•	36–47 meses;
•	48–59 meses;
•	60–71 meses.
Cada marco deve possuir:
•	domínio;
•	descrição;
•	faixa etária;
•	status;
•	observação opcional.
Estados:
•	alcançado;
•	não alcançado;
•	não observado.
O agente não deve inventar marcos clínicos. Os dados iniciais devem ser identificados como conteúdo provisório até revisão profissional.
________________________________________
11.8 Assistente científico
A IA deve responder somente com base nos documentos indexados.
Fluxo obrigatório:
pergunta
→ validação
→ criação do embedding
→ busca dos trechos relevantes
→ montagem do contexto
→ resposta do modelo
→ apresentação das fontes
A resposta deve conter:
•	síntese;
•	fontes;
•	instituição;
•	título do documento;
•	página ou trecho, quando disponível.
Caso não haja evidência suficiente:
Não encontrei informação suficiente na base científica cadastrada.
O sistema não deve completar a resposta com conhecimento externo sem informar.
Bloqueios:
•	diagnóstico;
•	prescrição;
•	dose de medicamento;
•	conduta emergencial individualizada;
•	emissão de laudo;
•	substituição da avaliação profissional.
Mensagem fixa:
Este recurso oferece apoio informativo com base nos documentos cadastrados e não substitui avaliação clínica.
________________________________________
11.9 Relatórios
O relatório deve ser gerado a partir dos dados da consulta.
Estrutura sugerida:
1.	Identificação da criança
2.	Data e idade
3.	Alimentação
4.	Sono
5.	Crescimento
6.	Desenvolvimento
7.	Vacinação
8.	Observações clínicas
9.	Conduta
Fluxo:
dados da consulta
→ geração inicial
→ revisão do profissional
→ edição
→ finalização
→ exportação em PDF
A IA nunca deve finalizar automaticamente o relatório.
O profissional deve revisar e confirmar.

13. Procedimento obrigatório por fase
Para cada fase, o agente deve seguir este ciclo:
1. Diagnóstico
Antes de alterar código:
•	verificar estrutura atual;
•	verificar branch;
•	verificar status do Git;
•	verificar dependências;
•	identificar arquivos envolvidos;
•	identificar riscos.
2. Plano
Descrever:
•	objetivo;
•	arquivos a criar;
•	arquivos a alterar;
•	modelo de dados;
•	riscos;
•	testes.
3. Implementação
Executar apenas o plano aprovado ou claramente necessário.
4. Validação
Executar:
npm run lint
npm run typecheck
npm test
npm run build
Quando houver E2E:
npm run test:e2e
5. Revisão
Verificar:
•	segurança;
•	isolamento entre usuários;
•	validação dos dados;
•	acessibilidade;
•	responsividade;
•	tratamento de erros;
•	ausência de segredos;
•	ausência de logs sensíveis.
6. Relatório
Entregar:
•	resumo;
•	arquivos alterados;
•	testes executados;
•	resultados;
•	pendências;
•	riscos;
•	próximos passos.
7. Git
Somente após testes aprovados:
git status
git diff --check
git add .
git commit -m "<tipo>: <descrição>"
git push
Nunca executar git push --force sem autorização explícita.

Usar Conventional Commits.

docs: documentar fluxo de geração de relatórios
refactor: separar serviço de pacientes
chore: configurar ambiente de testes
Cada commit deve representar uma unidade lógica concluída.
